# Agromonitoring — Система мониторинга сельхозформирований (Frontend)

Данный проект представляет собой Frontend-прототип (Single Page Application) для системы мониторинга применения удобрений и пестицидов в сельскохозяйственном секторе. Приложение предоставляет интерфейсы для расчета норм внесения, аналитики цепочек поставок, визуализации карт и генерации различных отчетов.

Текущий статус: Разработка интерфейсной части и роутинга завершена. Для автономного тестирования подключен локальный mock-сервер (json-server). Проект полностью готов к интеграции с реальным REST API.

---

## Технологический стек

- Сборка и ядро: React 18, Vite
- UI/UX Фреймворк: Ant Design (используются готовые паттерны форм, таблиц, модальных окон)
- Маршрутизация: React Router v6
- Графики и аналитика: Recharts
- Гео-информационная система: Leaflet, React-Leaflet
- Генерация документов: jspdf, docx, xlsx, file-saver
- Сетевые запросы: Axios (настроен глобальный перехватчик ошибок)

## Быстрый старт

Проект поддерживает два варианта запуска: изолированно через Docker (оптимально для бэкенд-разработчиков) или локально через Node.js.

### Вариант 1: Запуск через Docker (Frontend + Mock Backend)
Если у вас нет желания настраивать локальное окружение Node.js, достаточно выполнить одну команду в корне проекта:
```bash
docker-compose up --build
```
- Клиентская часть (Nginx) будет доступна по адресу `http://localhost:80`
- Временный mock-сервер API (`json-server`) поднимется на порту `http://localhost:3001`

### Вариант 2: Локальная разработка
Убедитесь, что у вас установлен Node.js версии 18 или выше.

```bash
# 1. Загрузите зависимости
npm install

# 2. Создайте файл конфигурации .env в корне проекта. Содержимое:
# VITE_API_BASE_URL=http://localhost:3001

# 3. Запустите mock-сервер бэкенда (в первом терминале)
npm run server

# 4. Запустите Frontend-сборщик (во втором терминале)
npm run dev
```

## Слой API (Интеграция с Backend)

Все сетевые запросы изолированы в директории `src/api/`. Компоненты приложения не делают прямых вызовов fetch или axios напрямую. Вместо этого они обращаются к сервисным методам:

```javascript
// Пример вызова из компонента:
import { dashboardApi } from '../api';
const data = await dashboardApi.getKPI();
```

При переходе на реальный бэкенд вам потребуется:
1. Заменить значение `VITE_API_BASE_URL` в файле `.env` на реальный адрес сервера.
2. При необходимости скорректировать пути эндпоинтов или ключи JSON-ответов внутри файлов папки `src/api/`.

Обработка ошибок: В проекте настроен глобальный перехватчик axios в файле `src/api/client.js`. При любой ошибке 400 или 500 со стороны бэкенда, фронтенд автоматически перехватит её и покажет пользователю аккуратное уведомление.

## Реализованные экраны

1. Login — Экран входа в систему. Тесно связан с глобальным AuthContext.
2. Dashboard — Аналитическая панель. Включает графики Recharts и карту Leaflet с полигонами земельных участков.
3. Calculator — Онлайн-калькулятор расчета норм внесения с учетом площадей и цен.
4. Documents — Модуль хранения и выгрузки файлов.
5. Reports — Табличное представление истории созданных отчетов.
6. References — Справочники номенклатуры системы.
7. Audit — Страница аудита безопасности (доступна только администраторам).

## Что ожидается от Backend-части

Текущий Frontend берет на себя всю визуализацию и роутинг, однако обработка сложной бизнес-логики остается зоной ответственности будущего сервера:

- Искусственный Интеллект (AI): Интеграция с языковыми моделями (LLM) для поиска фиктивных счетов-фактур должна происходить строго на сервере. Фронтенд просто ожидает готовый текстовый инсайт для отображения.
- Анализ ценообразования: Вычисление наценок в цепочках поставок реализуется на сервере, чтобы избежать утечек бизнес-логики на сторону клиента.
- Парсинг Excel: Обработка загружаемых счетов-фактур будет происходить на сервере (задача фронтенда показать интерфейс загрузки файла).
- Топология (GeoJSON): Для точной отрисовки границ участков сервер должен отдавать координаты в формате FeatureCollection.
- Матрица доступов (RBAC): Фронтенд визуально скрывает страницы в зависимости от роли (Admin, User, Employee), однако сервер должен обеспечивать строгую проверку прав на каждое изменение данных (POST/DELETE).
