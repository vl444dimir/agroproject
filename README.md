# 🌾 Agromonitoring — Система мониторинга сельхозформирований (Frontend MVP)

Проект представляет собой Frontend-прототип (Single Page Application) для мониторинга применения удобрений и пестицидов на сельскохозяйственных угодьях. Приложение обеспечивает интерфейсы для расчета норм, аналитики цепочек поставок, визуализации карт и генерации отчетности.

⚠️ **Текущий статус**: *Frontend-каркас завершен. Для имитации бэкенда интегрирован `json-server`. Проект готов к плавной интеграции с Production REST API сервером.*

---

## 🛠 Технологический стек

* **Сборка и Ядро**: React 18, Vite.
* **UI/UX Фреймворк**: Ant Design (готовые паттерны форм, таблиц, модальных окон).
* **Маршрутизация**: React Router v6.
* **Графики и Аналитика**: Recharts.
* **Гео-информационная система (ГИС)**: Leaflet, React-Leaflet. (WIP)
* **Генерация документов**: jspdf, docx, xlsx, file-saver.
* **Сетевое взаимодействие**: Axios (настроен глобальный интерцептор ошибок).

## 🚀 Быстрый старт

Проект поддерживает два варианта запуска: через Docker (рекомендуется для бэкендеров) или локально через Node.js.

### Вариант 1: Запуск через Docker (Frontend + Mock Backend)
Вам не нужен установленный Node.js. Достаточно выполнить одну команду в корне проекта:
```bash
docker-compose up --build
```
* **Frontend-клиент** (Nginx) будет доступен по адресу: `http://localhost:80`
* **Mock-сервер API** (`json-server`) будет поднят на порту: `http://localhost:3001`

### Вариант 2: Локальная разработка
Убедитесь, что у вас установлен **Node.js 18+**.
```bash
# 1. Установка зависимостей
npm install

# 2. Переменные окружения (Создайте файл .env)
# Содержимое .env: VITE_API_BASE_URL=http://localhost:3001

# 3. Запуск mock-сервера бэкенда (в первом терминале)
npm run server

# 4. Запуск Frontend-сборщика (во втором терминале)
npm run dev
```

## 🔌 Слой API (Интеграция с Backend)

Все сетевые запросы изолированы в директории `src/api/`. Компоненты приложения не делают прямых вызовов `fetch` или `axios.get(url)`. Вместо этого они вызывают сервисные методы:

```javascript
// Пример из компонентов:
import { dashboardApi } from '../api';
const data = await dashboardApi.getKPI();
```

При переходе на реальный бэкенд достаточно:
1. Изменить переменную `VITE_API_BASE_URL` в файле `.env`.
2. В случае необходимости, скорректировать пути эндпоинтов (endpoints) или ключи JSON-ответов внутри файлов папки `src/api/`.

> **UI Ошибки:** Настроен глобальный перехватчик `axios` в файле `src/api/client.js`. При любой ошибке 400/500 со стороны бэкенд-сервера фронтенд автоматически перехватит ошибку и отобразит красивое уведомление `notification.error` Ant Design.

## 🖥 Реализованные модули (Pages)

1. **`Login`** — Экран аутентификации пользователя (влияет на глобальный `AuthContext`).
2. **`Dashboard`** — Аналитический дашборд. Включает графики Recharts (топ удобрений/пестицидов) и карту `Leaflet` с полигонами геометрии.
3. **`Calculator`** — Онлайн-калькулятор расчета норм внесения с учетом площадей и цен.
4. **`Documents`** — Модуль хранения и выгрузки файлов (PDF, Excel, Word).
5. **`Reports`** — Табличное представление истории созданных отчетов.
6. **`References`** — Справочники системы.
7. **`Audit`** — Страница аудита системы безопасности (доступна только с ролью `admin`).

## 🏗 Зоны ответственности (Что ожидается от Backend)

Текущий Frontend-прототип готов отрисовывать сложную аналитику, однако непосредственно её процессинг является ответственностью будущего бэкенда:

* 🤖 **Искусственный Интеллект (AI)**: Интеграция с системами LLM (OpenAI/Gemini) для выявления фиктивных счетов-фактур должна происходить *строго на сервере*. Фронтенд ожидает получать готовый бизнес-инсайт текстом на отрисовку.
* 📊 **Анализ ценообразования**: Вычисление "наценок на каждом колене" цепочек поставок также реализуется на сервере для избежания утечек формул расчетов на клиенте.
* 📑 **Парсинг Excel (ЭСФ)**: Обработка загружаемых `*.xlsx` файлов счетов-фактур будет либо на клиенте через `SheetJS` на следующей итерации, либо напрямую на стороне парсера сервера. Во фронтенде предстоит создать лишь компонент `Upload`.
* 🗺 **GeoJSON-Топология**: Для рендера точных границ кадастровых участков районов бэкенд должен предоставлять данные в формате `FeatureCollection` (через PostGIS или Spatial Data). 
* 🔐 **Матрица Доступов (RBAC)**: Фронтенд сейчас умеет разделять маршруты (Админ, Пользователь, Сотрудник) при помощи Protected Routes, но бэкенд должен также обеспечить строгую проверку ролей сервера при любом POST/DELETE запросе.
