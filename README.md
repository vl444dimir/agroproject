# Agromonitoring Frontend (MVP)

Данный репозиторий содержит UI-прототип системы мониторинга сельскохозяйственных угодий. Проект подготовлен как SPA (Single Page Application) и готов к интеграции с полноценным бэкендом.

На данный момент для автономной разработки и тестирования интерфейсов используется мок-сервер (json-server).

## Технологический стек

- Костяк приложения: React 18, Vite
- UI: Ant Design
- Роутинг: React Router v6
- Графики: Recharts
- Работа с картами: Leaflet, React-Leaflet (WIP)
- HTTP клиент: Axios 
- Генерация файлов: jspdf, docx, xlsx, file-saver

## Запуск проекта

Проект можно запустить через Docker или локально. Рекомендуется использовать Docker, особенно если вы занимаетесь бекендом и не хотите настраивать среду Node.js.

### Вариант 1: Docker (Frontend + json-server)
Выполните в корне проекта:
```bash
docker-compose up --build
```
- Nginx с фронтендом поднимется на 80 порту (http://localhost)
- Сервер с заглушками (json-server) будет работать на 3001 порту (http://localhost:3001)

### Вариант 2: Локальная сборка
Убедитесь, что установлен Node.js версии 18 и выше.

1. Установите зависимости:
```bash
npm install
```

2. Подготовьте переменные окружения. Создайте в корне проекта файл `.env` и укажите путь до API:
```text
VITE_API_BASE_URL=http://localhost:3001
```

3. Для работы нужны два терминала:
В первом запустите мок-сервер:
```bash
npm run server
```
Во втором запустите фронтенд:
```bash
npm run dev
```

## Структура проекта

Код находится в папке `src/` и логически разделен на слои:

- `api/` — сервисный слой для работы с сетью. Все вызовы HTTP инкапсулированы здесь.
- `components/` — общие компоненты интерфейса (Layout, Header, Sidebar).
- `context/` — глобальный стейт приложения (преимущественно AuthContext для ролей и сессий).
- `pages/` — экраны-страницы (Dashboard, Calculator, Reports, Documents, Login, Audit).
- `router/` — настройки навигации и защита маршрутов (Protected Routes).

Взаимодействие с сервером настроено через Axios (`src/api/client.js`). На уровне клиента добавлен глобальный перехватчик: любая ошибка 400-500 со стороны API автоматически вызывает Toast-уведомление (notification) в интерфейсе.

## Чего ждет фронтенд от бэкенда

Текущий проект — это прототип (MVP), он берет на себя только визуализацию. Часть бизнес-требований из ТЗ не может быть реализована на статичном клиенте, поэтому они запланированы для реализации на стороне бэкенда:

1. ИИ и аналитика: поиск фиктивных ЭСФ и расчет маржинальности (наценок "на коленах") должен производиться на сервере. Фронтенд ждет готовых вычислений или инсайтов для отображения.
2. Обработка ЭСФ: парсинг и валидация входящих Excel-файлов пока отложена. В будущем фронтенд будет предоставлять компонент загрузки (Upload), а сервер (или клиентский SheetJS) будет конвертировать данные.
3. Геоданные: для полноценной отрисовки кадастровых участков поверх Leaflet сервер должен отдавать координаты в формате GeoJSON (FeatureCollection), так как хранить их цельно на клиенте слишком ресурсоемко.
4. Безопасность: матрица доступов (Admin, Employee, User) пока работает как визуальная заглушка (скрывает страницы). Реальная проверка прав на модификацию данных (POST/DELETE) должна происходить исключительно на стороне бэкенда.
